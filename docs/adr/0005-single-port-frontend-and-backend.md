# 0005. Single-port architecture: FastAPI serves both API and frontend bundle

## Status

Accepted — the deployed dashboard listens on port 8321 only (configurable via `DASHBOARD_PORT`). FastAPI serves both `/api/*` and the built Vite bundle from `dist/`. Documented in `SPEC.md` §2.1 and `CLAUDE.md`.

## Context

The dashboard is intended to be a *local operator console*. A user (or a fleet machine) starts it with `./start-dashboard.sh`, opens `http://localhost:8321`, and uses every tab from one browser context. The same binary runs in production via a systemd unit (`deploy/runner-dashboard.service`) on each fleet node.

Once the frontend migrated to Vite (see ADR 0001) two ports were briefly in scope:

- **Two ports (split):** Vite/Node serves the frontend on one port (5173 in dev), FastAPI serves `/api/*` on another (8321), and a reverse proxy or `fetch(...)` calls cross-origin to the API.
- **One port (current):** FastAPI serves both `/api/*` and the static Vite output from `dist/`. In dev, `npm run dev` runs Vite on 5173 and proxies `/api` back to 8321 (`vite.config.ts` `server.proxy`). In production, `npm run build` emits `dist/` and FastAPI mounts it as static files.

Considerations weighed:

- **Deployment surface.** Each fleet node already runs a systemd service for the dashboard. Adding a second service for the frontend doubles the unit count, the log destinations, the restart policies, and the health checks.
- **CORS and auth.** Two-origin setups must either send `Access-Control-Allow-Origin` everywhere (and reason about credentialed vs non-credentialed requests) or front both with a third reverse proxy. Single-origin removes the entire problem class.
- **PWA installability.** A PWA (manifest + service worker + offline shell, PR #268) is simpler when the bundle and the API live on the same origin — the service worker scope, navigation preload, and `start_url` all line up.
- **Operator UX.** The dashboard's selling point is "open the bookmark and everything is there". Two ports means two URLs, two TLS configs, and two firewall rules per node.
- **Tailscale Funnel and remote access.** The `docs/tailscale-funnel.md` design exposes the dashboard through a single hostname; that maps cleanly onto a single port.
- **Dev ergonomics.** Frontend developers want hot-module reload from Vite, but they also want to call real `/api/*` endpoints. Vite's dev-server proxy (`server.proxy: { '/api': 'http://localhost:5001' }` in `vite.config.ts`) gives both with a single `fetch('/api/...')` call from the browser.

## Decision

Run the dashboard on a **single port (8321)** for both `/api/*` and the frontend bundle.

Specifically:

1. **Production.** `npm run build` produces `dist/`. FastAPI mounts `dist/` as static files and serves `index.html` for any route that does not match an `/api/*` prefix (SPA fallback). The systemd unit binds only `0.0.0.0:8321` (or the configured `DASHBOARD_PORT`). No separate frontend service.
2. **Development.** Two ports *do* exist temporarily: `python server.py` on 8321 and `npm run dev` on 5173. The Vite config proxies `/api` from 5173 back to the backend, so frontend code always issues same-origin `fetch('/api/...')` calls regardless of mode. This is a dev-only seam, not a deployed surface.
3. **All cross-process communication is HTTP.** The frontend never imports backend code; the backend never reads the frontend's source. Their contract is the OpenAPI surface generated by FastAPI plus the dispatch envelope from ADR 0002.
4. **Cross-repo callers** (Maxwell tab → Maxwell-Daemon, fleet workflows in Repository_Management) hit the same `:8321` and discover the contract via `GET /api/version`.

## Consequences

**Easier:**

- One systemd unit, one log file, one port to firewall, one TLS termination, one Tailscale Funnel mapping per node.
- No CORS configuration is required for normal operation; same-origin requests "just work".
- The PWA service worker has a clean single-origin scope and can cache both `/api/*` responses and static assets under one strategy.
- Operator install is "open the bookmark" — no need to remember which port hosts the UI vs the API.
- The single-port assumption simplifies the deployment-drift detector (`backend/deployment_drift.py`) and the local-app monitoring view, which only need to probe one endpoint per node.

**Harder / cost:**

- The backend process is responsible for static-asset serving. It is fast enough today, but at much higher request volume a fronting reverse proxy (nginx, Caddy) might be needed; that proxy would *terminate* on 8321 toward FastAPI, preserving the contract.
- Frontend deploy is coupled to backend deploy: shipping a new bundle requires restarting the service. The deploy script (`deploy/update-deployed.sh`) handles this, and the rollback marker satisfies the "Reversible" engineering principle.
- Dev-mode requires two processes (`python server.py` + `npm run dev`) and a working proxy config; new contributors occasionally trip on running only one of them. The `start-dashboard.sh` script and the dev README mitigate this.
- A bug in `/api/*` and a bug in the bundle share the same blast radius — a crashing FastAPI process serves no UI either. The "Orthogonality" principle is preserved at the *route* level (one `/api/*` 5xx must not cascade into others) rather than at the process level.

If the dashboard ever needs to scale beyond a single-process operator console — for instance, a hosted multi-tenant variant — the right move is to terminate TLS and serve static assets at the edge (CDN or reverse proxy) while keeping FastAPI behind it, rather than splitting the codebase into two deployables.
